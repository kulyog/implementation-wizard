## AUD2-M01

| Field | Detail |
|---|---|
| **Finding ID** | AUD2-M01 |
| **Severity** | 🟠 Major |
| **Category** | Code Structure — Partial Remediation Created Ambiguous Split-Folder State |
| **File** | `/src/utils/stepLogic.js` and `/src/store/exportImport.js`, `/src/store/integrityCheck.js`, `/src/store/migration.js`, `/src/store/schema.js` |
| **Specification Reference** | 02-technical-specifications.md § 5 (File and Folder Structure) |
| **Origin** | New finding — introduced by partial resolution of AUD-M01 |

**Description:**

In v1.0, all utility files were incorrectly located in `/src/store/`. The Owner accepted this deviation as-is. However, when `stepLogic.js` was confirmed and the `/src/utils/` directory was made explicit in the new folder listing, the result is a codebase that now has *two* utility-like directories with an undocumented, arbitrary split:

- `/src/utils/` — contains only `stepLogic.js`
- `/src/store/` — contains `exportImport.js`, `integrityCheck.js`, `migration.js`, `schema.js`

There is no logical basis for this split. All five files are pure utility/logic modules. None of them manages application state in the Redux/Zustand sense that the `/store/` naming convention implies. The result is worse than the original deviation:

- In v1.0 a developer could find all utility files by checking `/src/store/` — wrong name, but predictable.
- In v2.0, a developer or Claude Code session must know to check *both* directories to find all utility functions. There is no discoverable rule for which file belongs where.
- Any Claude Code prompt that says "update the import validation logic" will search `/src/utils/` first (the semantically correct location), find only `stepLogic.js`, and either stop or hallucinate a path. The actual file is in `/src/store/`.

The 6 new Vitest tests that resolved AUD-M06 almost certainly import from `/src/utils/stepLogic.js`. The existing 19 `exportImport.test.js` tests import from `/src/store/exportImport.js`. Both test files pass — but they import from different utility directories. This is an invisible inconsistency that will compound over time.

**Risk:** Every future Claude Code session that touches utility logic has a 50% chance of looking in the wrong folder for a given file. This is a cognitive overhead and regression risk that compounds with each V1.1 development session. The accepted-as-is decision on AUD-M01 is respected, but the new structural ambiguity it has created cannot be accepted without documentation.

**Recommended Fix:**

The Auditor does not require renaming at V1.0 — the acceptance decision stands. But two mitigations are mandatory:

1. Add the following to `CLAUDE.md`, explicitly:
   ```
   IMPORTANT — Utility File Locations (non-standard):
   /src/utils/stepLogic.js — business rule functions only
   /src/store/ — exportImport.js, integrityCheck.js, migration.js, schema.js
   When working on export/import or data integrity, look in /src/store/, NOT /src/utils/.
   ```
2. Add a comment at the top of `stepLogic.js`: `// See also: /src/store/ for related utility files (exportImport, migration, integrityCheck)`

If these mitigations are not in place, future Claude Code sessions will create structural regressions.

## AUD2-M02

| Field | Detail |
|---|---|
| **Finding ID** | AUD2-M02 |
| **Severity** | 🟠 Major |
| **Category** | Deployment — Sprint 3 Exit Criterion Not Satisfied |
| **File** | `vite.config.js`, `gh-pages` branch |
| **Specification Reference** | 02-technical-specifications.md Sprint 3 exit criteria, S3-12, S3-13, C-09 |
| **Origin** | Carried from AUD-M05 — build-level fix confirmed, deployment evidence absent |

**Description:**

AUD-M05 required two things: (1) adding `base: '/implementation-wizard/'` to `vite.config.js`, and (2) confirming a live GitHub Pages URL that loads and functions correctly. The Owner has confirmed (1). No evidence of (2) has been provided.

The Sprint 3 exit criteria state unambiguously:
- *"Application is deployed to GitHub Pages with a live URL"* (primary criterion)
- *"Post-deploy smoke test — open in clean browser, create project, complete steps 1–3"* (S3-13)

The `/dist` folder being ready is a necessary precondition for deployment. It is not deployment itself. A `vite build` that passes locally with a correct `base` path is a correct build artefact — it does not prove that the application is accessible, loads correctly, or that the `gh-pages` branch push was executed without error.

Common failure modes at this stage that would not be caught by the build:
- The GitHub repository is private, making the Pages URL inaccessible
- The `gh-pages` branch was not pushed (only the `base` path was updated locally)
- The GitHub Pages deployment is configured to serve from `main` or `docs/`, not `gh-pages`
- The base path in `vite.config.js` does not match the actual repository name

None of these failure modes are detectable from the information provided. The application is not deployed until a URL exists and has been smoke-tested per S3-13.

**Risk:** Step 23 of the 24-step process (Deploy / Publish) cannot be marked Complete without a live URL. If deployment is assumed complete without verification, the Owner will believe the project is finished when it is not. More practically, the sole backup mechanism (export) is not useful if the application cannot be opened by the user.

**Recommended Fix:**

1. Execute the `gh-pages` deployment:
   ```bash
   npm run build
   npx gh-pages -d dist
   ```
   Or push the `/dist` contents to the `gh-pages` branch manually.
2. In the GitHub repository settings, confirm GitHub Pages is set to deploy from the `gh-pages` branch, root directory.
3. Wait for the Pages deployment to complete (typically 1–3 minutes).
4. Open `https://[username].github.io/implementation-wizard/` in a clean browser (no localhost, no cache).
5. Create a project, navigate steps 1–3, verify localStorage persistence.
6. Record the live URL. Mark Step 23 Complete only after this verification.

## AUD2-N01

| Field | Detail |
|---|---|
| **Finding ID** | AUD2-N01 |
| **Severity** | 🟡 Minor |
| **Category** | Code Structure — Inconsistent `/shared/` Subdirectory |
| **File** | `/src/components/shared/ErrorBoundary.jsx` vs `/src/components/ConfirmModal.jsx`, `StatusBadge.jsx`, `ActorBadge.jsx`, `Toast.jsx` |
| **Specification Reference** | 02-technical-specifications.md § 5 (Component structure), 04-wireframes.md (Screen Inventory) |

**Description:**

The addition of `ErrorBoundary.jsx` into `/src/components/shared/` has created a partially implemented subdirectory. The specification defines `/components/shared/` as the home for: `StatusBadge.jsx`, `ActorBadge.jsx`, `ConfirmModal.jsx`, and `Toast.jsx`. All four of these remain in the flat `/src/components/` root. `ErrorBoundary.jsx` is the only component correctly placed in `/shared/`.

The result is that `/shared/` exists but contains only 1 of the 5 components that belong there. Any import statement referencing `from '../shared/StatusBadge'` will fail — those components are actually in `'../StatusBadge'`. A developer finding `ErrorBoundary.jsx` in `/shared/` will reasonably look for other shared components there and not find them.

**Risk:** Low — does not affect the build or runtime behaviour. Will generate incorrect import path suggestions in Claude Code sessions that notice the `/shared/` subdirectory exists.

**Recommended Fix:** Either (a) move all four remaining shared components (`StatusBadge`, `ActorBadge`, `ConfirmModal`, `Toast`) into `/src/components/shared/` to complete the structure, or (b) move `ErrorBoundary.jsx` back to the flat `/src/components/` root to eliminate the incomplete subdirectory. Option (a) is the correct long-term approach and resolves this finding cleanly.

## AUD2-N02

| Field | Detail |
|---|---|
| **Finding ID** | AUD2-N02 |
| **Severity** | 🟡 Minor |
| **Category** | Architecture Deviation — AppShell Logic Embedded in App.jsx |
| **File** | `/src/App.jsx` (lines 59–104 per Owner's confirmation) |
| **Specification Reference** | 02-technical-specifications.md § 5 (File Structure), § 8 (Component Hierarchy), 04-wireframes.md (Global Shell) |

**Description:**

AUD-C01 has been accepted as a clarification rather than a fix: the AppShell functionality (navigation bar, export button, import button, breadcrumb) is implemented directly in `App.jsx` lines 59–104, rather than as a standalone `AppShell.jsx` component. The Auditor accepts this as functionally equivalent for V1.0.

However, the specification designates `AppShell` as a separate component for a reason. The intent is separation of concerns: `App.jsx` is the view router (it owns `currentView` and `currentProjectId` state), not the layout shell. Embedding approximately 46 lines of layout JSX inside the root component means:

- `App.jsx` now has two responsibilities: routing and shell rendering
- Future changes to the navigation bar require editing the root component — the highest-risk file in the codebase
- The `AppShell` component referenced in `CLAUDE.md` (if it exists) will not match what the codebase actually contains, creating confusion in future Claude Code sessions

**Risk:** Low for V1.0. Increases slightly for V1.1 when the Open in Claude deep-link (C-03) and Markdown export (C-05) are added — both of which are AppShell-level controls that would require modifying `App.jsx`.

**Recommended Fix:** In V1.1, extract the navigation bar JSX from `App.jsx` lines 59–104 into a dedicated `AppShell.jsx` component. Pass `currentView`, `onNavigate`, `exportData`, and `importData` as props. This is a 1-hour refactor that aligns the codebase with the specification and reduces `App.jsx` complexity. Update `CLAUDE.md` to reflect the actual current state (AppShell in App.jsx) to prevent confusion until then.
